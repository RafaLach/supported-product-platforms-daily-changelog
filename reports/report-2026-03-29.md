## New Coverage Required

- **[AISPM] [HIGH] Microsoft Foundry IQ (Preview)**: New knowledge-base-connected agent feature in Foundry Agent Service. Foundry IQ enables agents to query connected knowledge bases, introducing new entity types: IQ-enabled agents, knowledge base connections, and IQ query sessions. AISPM must discover and inventory Foundry IQ agents and their knowledge base bindings to map data access patterns and ensure sensitive knowledge bases are governed. Source: https://learn.microsoft.com/en-us/azure/foundry/agents/concepts/what-is-foundry-iq

- **[AISPM] [HIGH] Microsoft Agent Framework**: New comprehensive agent orchestration framework with support for 11+ inference providers (Foundry, Azure OpenAI, OpenAI, Anthropic, Amazon Bedrock, GitHub Copilot, Ollama) and A2A (Agent-to-Agent) protocol for cross-agent communication. Introduces new entity types: framework-managed agents, multi-provider agent configurations, A2A proxy agents, and hosted agent deployments. AISPM must discover agents built with this framework across all supported providers and map their tool bindings, MCP connections, and inter-agent communication patterns. Source: https://learn.microsoft.com/en-us/agent-framework/agents/

- **[AISPM] [HIGH] Power Apps "Vibe" AI Copilot Experience**: New AI-native app creation experience in Power Apps (preview). Users describe business problems in natural language, and Vibe generates complete solutions including data models, business logic, and UI code. Creates new entity types (Vibe-generated apps, AI-created data models and code components) that need discovery and inventory tracking. AISPM must map Vibe apps as a distinct resource type — they may contain AI-generated code with different security characteristics than manually built apps. Source: https://learn.microsoft.com/en-us/power-apps/vibe/overview

- **[AISPM] [HIGH] Power Pages Agent Enablement**: New capability to embed AI agents within Power Pages sites. Agents automatically gain site-context awareness and can generate responses based on visitor web roles. Introduces agent entities deployed on public-facing web pages with role-based access control. AISPM must discover these agents and AIDR must monitor their data flows given the external exposure surface — agents responding to unauthenticated or low-privilege visitors pose data leakage risk. Source: https://learn.microsoft.com/en-us/power-pages/getting-started/enable-agent

- **[AISPM] [HIGH] Microsoft Foundry Hosted Agents**: New managed deployment model for AI agents in Foundry Agent Service. Agents can be deployed as hosted services with lifecycle management (create, deploy, scale, stop). Introduces new infrastructure entities: hosted agent instances, deployment configurations, and service endpoints. AISPM must discover hosted agents and map their model bindings, tool access, and network exposure. Source: https://learn.microsoft.com/en-us/azure/foundry/agents/quickstarts/quickstart-hosted-agent

- **[AISPM] [MEDIUM] Fireworks Models on Microsoft Foundry (Preview)**: New third-party model provider (Fireworks AI) integrated into Microsoft Foundry. Expands the model catalog with Fireworks models. AISPM model inventory must be updated to discover and track deployments using Fireworks models for compliance and governance. Source: https://learn.microsoft.com/en-us/azure/foundry/foundry-models/how-to/fireworks/enable-fireworks-models

- **[AISPM] [MEDIUM] Foundry Tools (Rebranded Azure AI Services)**: "Foundry Tools" is the new name for Azure AI Services within the Microsoft Foundry platform. Includes Azure OpenAI, Content Safety, Speech, Vision, Document Intelligence, Language, and Translation services. While the underlying services exist, the rebranding under Foundry means new API endpoints, SDK paths, and resource identifiers that AISPM inventory mappings must be updated to track. Source: https://learn.microsoft.com/en-us/azure/ai-services/what-are-ai-services

- **[AISPM] [MEDIUM] Power Apps Code Apps**: New code-first app creation type in Power Apps. Developers build with React/Vue locally and deploy to Power Platform via `pac code push`. These apps have different security characteristics than canvas/model-driven apps — developer-authored code runs within Power Platform's runtime. AISPM inventory should track this new app type distinctly. Source: https://learn.microsoft.com/en-us/power-apps/developer/code-apps/overview

- **[ENDPOINT] [MEDIUM] Windsurf SKILL.md and Enterprise MDM Skills**: Windsurf added SKILL.md file support for custom skill definitions (`.windsurf/skills/`) and MDM-managed system-level skill definitions for Enterprise deployments. Skills can bundle supporting files and define multi-step workflows that Cascade AI executes. Enterprise MDM integration enables organization-wide skill policies deployed to all user machines — endpoint monitoring should track skill definitions and their tool/MCP interactions. Source: https://docs.windsurf.com/windsurf/cascade/skills

- **[AIRT] [MEDIUM] Claude Code Integration with Microsoft Foundry**: New documentation for configuring Claude Code as an AI development tool connected to Microsoft Foundry. This creates a new attack surface where Claude Code agents can interact with Foundry resources, models, and deployed agents. AIRT should test for prompt injection and privilege escalation in Claude Code sessions connected to Foundry. Source: https://learn.microsoft.com/en-us/azure/foundry/foundry-models/how-to/configure-claude-code
