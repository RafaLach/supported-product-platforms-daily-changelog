## New Coverage Required

- **[AISPM] [HIGH] AWS Agent Registry**: Entirely new AWS service (April 9, 2026) that acts as the organization-wide catalog for agents, MCP servers, tools, agent skills, and custom resources — deployed inside Amazon Bedrock AgentCore. Introduces multiple new inventory entities that Zenity must discover and map: **Registries** (the catalog itself, optionally scoped by resource type, stage, or business unit), **Registry Records** (individual agent / MCP / tool / skill entries with protocol-validated metadata), **publisher / curator / approval-workflow roles**, and registry-level authorization configuration (IAM or JWT from the customer's IdP). Because the registry exposes an **MCP-native endpoint**, any MCP-compatible agent (Claude, Cursor, Kiro, Amazon Q CLI, Bedrock AgentCore Runtime) can discover and invoke the listed tools — creating a new lateral-movement and shadow-tool risk surface where an overbroad approval setting or an unreviewed record effectively grants enterprise-wide agent access. AISPM should ingest registry contents via the Registry service APIs, correlate records against their downstream runtime deployments, flag records that bypass approval, and track `CloudTrail` events for record publication, approval, and deprecation. Source: https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/registry.html

- **[AIRT] [MEDIUM] AWS Agent Registry approval bypass and MCP discovery surface**: Because AWS Agent Registry is queryable over an MCP endpoint and supports "custom resources" with flexible metadata, AIRT red-teaming scenarios must cover (a) agents discovering and calling MCP tools they were never explicitly granted via a single MCP query against the registry, (b) poisoned record metadata (description, schema) used as a prompt-injection vector against AI consumers of the registry, and (c) unapproved/draft records that leak through misconfigured approval settings. Source: https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/registry.html

- **[ENDPOINT] [MEDIUM] Cursor CLI Debug Mode (autonomous hypothesis-driven debugging)**: Cursor CLI 0.122.x (April 14, 2026) introduces `/debug`, a self-directed debugging loop where the agent generates hypotheses, edits the codebase to add log statements, runs the program, and iterates until it pinpoints a root cause before issuing a fix. This is a new local-agent capability that writes to files and executes code repeatedly without per-step user approval. `/btw` also allows out-of-band side queries that may read unrelated project files mid-task. ENDPOINT monitoring must inventory CLI sessions that invoke `/debug` (potentially large diff surface from log insertions), track `/statusline` custom scripts as a potential code-exec surface, and flag `/config` changes that switch underlying model providers. Source: https://cursor.com/docs/agent/debug-mode
